git# План практики (после прочтения теории)

## Главный принцип
**Комбинировать, но центр тяжести — свой фреймворк, который ты строишь С НУЛЯ сам.**

- Только задачки — мало: они тренируют лишь live-coding, а собес AQA крутится вокруг
  «расскажи про свой фреймворк».
- Готовый `framework/` из набора — это РЕФЕРЕНС и страховка. Подсматривай, когда застрял,
  но печатай руками сам. Цель — уметь объяснить КАЖДУЮ строчку.

## Ежедневный ритм (~1.5–2 часа)
1. **Разминка 20–30 мин** — 1–2 задачки (Codewars 8→6 kyu). Раз в пару дней — одна
   «что выведется» из tasks/python-tricky.md.
2. **Основной блок 1–1.5 ч** — строишь фреймворк по этапам (ниже).
3. **Фоном 15 мин** — SQL (sql-ex.ru) + пара Linux-команд в терминале.

## Поэтапная постройка фреймворка (2–3 недели)

Делай новый пустой проект `my-aqa-framework`, НЕ копируй готовый. Каждый этап = рабочий
коммит в git. К каждому этапу указан блок теории для опоры.

### Этап 1. Каркас (день 1–2) — блоки 01, 02
- [ ] `python -m venv venv`, активировать
- [ ] `requirements.txt`: pytest, selenium, requests
- [ ] структура папок: `pages/`, `tests/ui/`, `tests/api/`
- [ ] `git init`, `.gitignore` (venv, __pycache__, allure-results)
- [ ] первый тест `test_smoke.py` с `assert 2+2==4` — убедиться, что `pytest` запускается
- [ ] `pytest.ini` с маркерами smoke/regression/ui/api
- **Проверка:** `pytest -v` зелёный.

### Этап 2. Первый UI-тест «в лоб» (день 3–4) — блок 03
Сначала БЕЗ Page Object — чтобы прочувствовать боль и понять, зачем POM.
- [ ] `conftest.py`: фикстура `driver` (`yield` + `driver.quit()`)
- [ ] headless через `os.getenv("HEADLESS")`
- [ ] тест логина на the-internet.herokuapp.com с локаторами прямо в тесте
- [ ] заменить любые `time.sleep` на `WebDriverWait` + `expected_conditions`
- **Проверка:** тест логина проходит, второй раз не флачит.

### Этап 3. Рефакторинг в Page Object (день 5–6) — блок 04, 20
- [ ] `BasePage` (open, find, click, type, text_of на explicit wait)
- [ ] `LoginPage`, `SecurePage` — локаторы как атрибуты, действия как методы
- [ ] переписать тест: в нём НЕТ локаторов, только бизнес-шаги
- [ ] параметризация негативных кейсов (`@pytest.mark.parametrize`)
- **Проверка:** сравни тест до/после — почувствуй, насколько читаемее. Это твой главный
  рассказ на собесе.

### Этап 4. API-тесты (день 7) — блоки 05, 16
- [ ] фикстура `api` (requests.Session, scope=session) в conftest
- [ ] позитив: GET список, POST создание на reqres.in
- [ ] негатив: 404 на несуществующем
- [ ] (плюс) валидация схемы через pydantic/jsonschema
- **Проверка:** `pytest tests/api -v` зелёный.

### Этап 5. Отчётность и стабильность (день 8–9) — блоки 10, 09
- [ ] Allure: `@allure.epic/feature/story`, `@allure.step`
- [ ] хук `pytest_runtest_makereport` — скриншот при падении в Allure
- [ ] специально сломай локатор → убедись, что скриншот прикрепился → почини
- [ ] параллельный запуск: `pytest -n auto` (pytest-xdist)
- **Проверка:** `allure serve allure-results` показывает дерево с шагами.

### Этап 6. CI (день 10) — блоки 06, 12
- [ ] репозиторий на GitHub (или GitLab)
- [ ] `.github/workflows/tests.yml` (или `.gitlab-ci.yml`)
- [ ] API-тесты + UI в headless на каждый push
- [ ] зелёный пайплайн + бейдж в README
- **Проверка:** push → зелёная галка в Actions.

### Этап 7. Расширение (ongoing) — закрепление
- [ ] ещё страницы/сценарии the-internet: checkboxes, dropdown, dynamic_loading, frames, hovers, alerts
- [ ] генерация данных через Faker
- [ ] логирование (модуль logging)
- [ ] README: стек, как запустить, архитектура (это твоя витрина)

## После сборки — самое важное
**Научись рассказывать про фреймворк вслух** (см. блок 07 и behavioral/):
- стек и почему такой;
- архитектура: POM, фикстуры в conftest, разделение ui/api;
- что решает: стабильность (explicit wait), переиспользование (BasePage), отчётность
  (Allure + скриншоты), скорость (параллель), регресс (CI);
- покажи код и зелёный пайплайн.

Прогони это вслух 3–4 раза, будто на собесе.

## Типичные ошибки на этом этапе
- Копировать готовый фреймворк вместо ручной сборки → не сможешь объяснить.
- Уйти в задачки с головой, забив на фреймворк → провалишь главный вопрос.
- Делать всё идеально и долго на одном этапе → лучше рабочий минимум, потом улучшать.
- Не запускать вслух рассказ про проект → на собесе «поплывёшь».
